Plan → trim → select: a deep-dive evidence loop

2026-05-16 · Design preview, not yet implemented · Probe results in .runs/paragraph_clip_probe_GPR75.json

Three jobs, three actors, two models — no new tools required:

  1. Sonnet (A1) plans the searches using only what we already have (gene_lookup, evidence_retrieval, gene_literature). The agent emits a structured search plan, not free-form tool calls.
  2. Orchestrator executes the plan + Haiku trims the results. All planned searches run in parallel; for each paper that comes back, Haiku reduces the paragraph/snippet pool to the load-bearing clips with one-line "why kept" reasons.
  3. Sonnet selects by clip ID. The trimmed clips come back tagged with stable IDs. Sonnet's only output is a list of clip_id picks plus their classifications — no quote field, no prose retyping. The orchestrator fills EvidenceClaim.quote from the clip's verbatim text.

The loop has one decision point: after seeing trimmed clips, Sonnet either selects + finishes, or plans more searches and goes around again. A1 and A2 run this loop independently against the shared clip pool (A1 selects surface-evidence clips, A2 selects biology-context clips); their selections merge in the ledger.

The verbatim-without-retyping property

Sonnet's selection output is the only place a quote could be paraphrased — and there's no quote field on it. The agent references the clip by ID, the orchestrator looks up the verbatim text in the clip pool, and the assembled EvidenceClaim.quote is byte-identical to the source body. The substring check at promotion passes by construction. Sonnet still writes the claim narrative (the agent's interpretation, which is meant to be synthesis, not a quote) — that field is not anchored.

How "needs more searches" works

Step 4 lets Sonnet emit needs_more_searches: true with an additional SearchPlan instead of selections. The orchestrator runs the new plan, Haiku trims the new papers, and the menu hands back to Sonnet for the actual selection pass. This is the iterative escape hatch for cases where:

  • The initial trim returned thin coverage of a category Sonnet thinks is load-bearing
  • A kept clip cites a paper that needs fetch_fulltext to verify
  • Sonnet realizes after seeing the trim that a different topic anchor would help (e.g., a paper mentioned a partner protein the planner didn't anticipate)

Cap at 2-3 plan iterations so the loop terminates.

Open questions for the design

  • A1 vs A2 split. Keep them parallel (each runs the loop independently against the shared clip pool, selecting their respective claim_types) or merge into one Sonnet agent that handles both? Two passes give the synthesizer independent reads; one pass is simpler.
  • Haiku batch size per paper. One Haiku call per paper is clean, but for a 100-paragraph paper that's 100 clips to judge in one prompt. Cap at ~50 per call? Multiple calls per paper?
  • The claim narrative field on Sonnet's selection. Today this is required (the agent's interpretation of the evidence). Keep it required, allow optional, or auto-generate from the clip text + classifications? My read: keep required — the agent's synthesis adds real signal beyond the verbatim quote.
  • Plan iteration cap. 2? 3? Need data from the first real run.
  • Trim model. Haiku at this step looked clean on the GPR75 probe; if quality demands surface (the LGR4 over-inclusion case), bump to Sonnet for the trim too. Currently Sonnet-everywhere is the conservative quality-led choice; Haiku-for-trim is the leaner default.
